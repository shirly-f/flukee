# Document Management System — Supabase Design

## Overview

A document management system that allows coaches to upload documents once and assign them to one or many trainees. Uses Supabase Storage (private bucket) for files and Supabase PostgreSQL for metadata. Row Level Security ensures coaches only manage their documents and trainees only access documents assigned to them.

---

## 1. Architecture

```
┌─────────────────┐     upload      ┌──────────────────────┐

└────────┬────────┘                 └──────────┬──────────┘
         │                                      │

         ▼                                      ▼
┌─────────────────┐                 ┌──────────────────────┐

│  (Trainee)      │   fetch signed   │                      │
└─────────────────┘                 └──────────────────────┘
```

## 2. Supabase Storage

### 2.1 Private Bucket

| Setting | Value |
|---------|-------|
| **Bucket name** | `trainee-documents` |
| **Public** | No (private) |
| **File size limit** | 10 MB |
| **Allowed MIME** | `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `text/plain` |

### 2.2 Object Path Structure

```
trainee-documents/
├── {coach_id}/
│   ├── {document_id}/
│   │   └── {unique_id}_{original_filename}
```

Example: `trainee-documents/coach_1/doc_abc123/1739123456_guide.pdf`

- **coach_id**: Owner; used for RLS and organization
- **document_id**: UUID from `documents` table; links storage to DB row
- **unique_id**: Timestamp to avoid collisions if same file uploaded twice
- **original_filename**: Sanitized original name for display

### 2.3 Storage RLS (Policies)

| Policy | Allowed |
|--------|---------|
| **Coaches INSERT** | `auth.uid() = coach_id` (from path) — upload only to own folder |
| **Coaches SELECT** | `auth.uid() = coach_id` |
| **Coaches DELETE** | `auth.uid() = coach_id` |
| **Trainees SELECT** | Via signed URL only; no direct storage SELECT |

*Note: Supabase Storage RLS is path-based. With `{coach_id}/` in the path, we enforce coach ownership. Trainees never access storage directly; they use signed URLs generated by the backend.*

## 3. PostgreSQL Schema

### 3.1 Users Table (Existing / Supabase Auth)

Assume `auth.users` exists (Supabase Auth) or a custom `users` table with `id`, `email`, `role` (`coach` | `trainee`), `name`.

**Flukee compatibility:** If `users.id` is `TEXT` (e.g. `coach_1`), use `TEXT` for `documents.coach_id` and `document_assignments.trainee_id` instead of `UUID`.

### 3.2 Documents Table

```sql
CREATE TABLE documents (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  coach_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  original_filename TEXT NOT NULL,
  storage_path TEXT NOT NULL UNIQUE,
  mime_type TEXT,
  file_size_bytes BIGINT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_documents_coach ON documents(coach_id);
CREATE INDEX idx_documents_created ON documents(created_at DESC);
```

| Column | Type | Description |
|--------|------|-------------|
| id | UUID | Primary key; used in storage path |
| coach_id | UUID | Uploading coach; ownership |
| title | TEXT | Display name (e.g. "Mindfulness Guide") |
| original_filename | TEXT | Original file name for downloads |
| storage_path | TEXT | Full path in bucket (e.g. `coach_1/doc_abc/1739_guide.pdf`) |
| mime_type | TEXT | e.g. `application/pdf` |
| file_size_bytes | BIGINT | File size |
| created_at, updated_at | TIMESTAMPTZ | Audit fields |

### 3.3 Document Assignments Table (Many-to-Many)

```sql
CREATE TABLE document_assignments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  trainee_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  assigned_at TIMESTAMPTZ DEFAULT NOW(),
  assigned_by UUID REFERENCES users(id),
  UNIQUE(document_id, trainee_id)
);

CREATE INDEX idx_assignments_document ON document_assignments(document_id);
CREATE INDEX idx_assignments_trainee ON document_assignments(trainee_id);
```

| Column | Type | Description |
|--------|------|-------------|
| id | UUID | Primary key |
| document_id | UUID | Document reference |
| trainee_id | UUID | Assigned trainee |
| assigned_at | TIMESTAMPTZ | When assigned |
| assigned_by | UUID | Coach who assigned |
| UNIQUE(document_id, trainee_id) | — | One assignment per document per trainee |

### 3.4 Optional: Task Link

If documents are used in tasks (read_document):

```sql
ALTER TABLE document_assignments
  ADD COLUMN task_id UUID REFERENCES tasks(id);
```

This links an assignment to a specific task. For pure document management without tasks, `task_id` can be NULL.

## 4. Row Level Security (RLS)

### 4.1 Documents Table

```sql
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

-- Coaches: full access to own documents
CREATE POLICY "Coaches manage own documents"
  ON documents FOR ALL
  USING (coach_id = auth.uid())
  WITH CHECK (coach_id = auth.uid());

-- Trainees: no direct access; use API with signed URLs
-- (Trainees don't SELECT from documents; backend generates signed URLs)
```

**Custom Auth:** If using custom auth (not Supabase Auth), create a `current_user_id()` function that returns the authenticated user ID from your JWT/session, and use it instead of `auth.uid()` in all policies.

### 4.2 Document Assignments Table

```sql
ALTER TABLE document_assignments ENABLE ROW LEVEL SECURITY;

-- Coaches: manage assignments for their documents only
CREATE POLICY "Coaches manage assignments for own documents"
  ON document_assignments FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM documents d
      WHERE d.id = document_assignments.document_id
      AND d.coach_id = auth.uid()
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM documents d
      WHERE d.id = document_assignments.document_id
      AND d.coach_id = auth.uid()
    )
  );

-- Trainees: read-only access to their assignments
CREATE POLICY "Trainees view own assignments"
  ON document_assignments FOR SELECT
  USING (trainee_id = auth.uid());
```

## 5. Signed URLs

### 5.1 Flow

1. Trainee requests document access (e.g. via task or document list).
2. Backend checks:
   - Trainee is authenticated.
   - Document is assigned to that trainee (via `document_assignments`).
3. Backend calls Supabase Storage:
   ```js
   const { data } = await supabase.storage
     .from('trainee-documents')
     .createSignedUrl(document.storage_path, 3600); // 1 hour
   ```
4. Backend returns `{ url: data.signedUrl }` to the client.
5. Client opens the URL (browser or `Linking.openURL` on mobile).

### 5.2 Expiration

- Typical: 1 hour (3600 seconds).
- Configurable via env: `SUPABASE_SIGNED_URL_EXPIRY_SECONDS`.

## 6. API Design

### 6.1 Coach Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/coaches/documents` | Upload document (multipart); create `documents` row and storage object |
| GET | `/api/coaches/documents` | List coach's documents |
| GET | `/api/coaches/documents/:id` | Get document metadata |
| PATCH | `/api/coaches/documents/:id` | Update title |
| DELETE | `/api/coaches/documents/:id` | Delete document and storage object; cascade assignments |
| POST | `/api/coaches/documents/:id/assign` | Assign to trainee(s): `{ traineeIds: string[] }` |
| DELETE | `/api/coaches/documents/:id/assign/:traineeId` | Unassign |

### 6.2 Trainee Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/trainees/documents` | List documents assigned to trainee |
| GET | `/api/trainees/documents/:id/access` | Get signed URL for document (verify assignment) |

### 6.3 Upload Payload (Coach)

```
POST /api/coaches/documents
Content-Type: multipart/form-data

file: <binary>
title: "Mindfulness Guide"
```

Response:
```json
{
  "id": "uuid",
  "title": "Mindfulness Guide",
  "originalFilename": "guide.pdf",
  "storagePath": "coach_1/uuid/1739_guide.pdf",
  "createdAt": "2025-02-11T..."
}
```

### 6.4 Assign Payload

```
POST /api/coaches/documents/:id/assign
Content-Type: application/json

{
  "traineeIds": ["trainee_1", "trainee_2"]
}
```

## 8. Integration with Existing Tasks

### Option A: Replace task metadata documentUrl

For `read_document` tasks:

- `metadata.documentUrl` → `metadata.documentId`
- When creating a read_document task, coach selects a document (from their library) and optionally assigns it in one step.
- Trainee fetches signed URL via `GET /api/trainees/documents/:documentId/access` or `GET /api/trainees/tasks/:taskId/document-access`.

### Option B: Keep both

- Standalone documents (upload → assign to trainees).
- Tasks can reference a document by `documentId` for read_document type.

## 9. Migration / Setup Steps

1. Create Supabase project.
2. Create private bucket `trainee-documents` and configure Storage RLS.
3. Run document and assignment table migrations in Supabase SQL editor.
4. Configure env vars in the backend.
5. Implement upload, assign, and signed-URL endpoints.

---

## 10. Supabase Auth vs Custom Auth

- **Supabase Auth**: Use `auth.uid()` in RLS; user IDs must match Supabase Auth user IDs.
- **Custom Auth (e.g. Flukee's token-based auth)**: Use the service role key for backend operations and enforce access in your API layer. RLS can be relaxed or you can create a `current_user_id()` function that reads from `request.jwt_claim('sub')` or a similar mechanism if you pass JWT to Supabase.
